# Commit 信息生成规范

## 规范要求 (Angular + Gitmoji 混合风格)

### 1. 格式结构

`<type>(<scope>): <emoji> <subject>`

- `scope` 是可选的，如果更改涉及特定模块请填写，否则省略括号
- `subject` 使用英文，动词开头，小写

### 2. Type 与 Emoji 的对应关系

- `feat` -> ✨ (新功能)
- `fix` -> 🐛 (修复 Bug)
- `docs` -> 📝 (文档变动)
- `style` -> 💄 (格式/样式调整，不影响逻辑)
- `refactor` -> ♻️ (重构)
- `perf` -> ⚡️ (性能优化)
- `test` -> ✅ (测试相关)
- `chore` -> 🔧 (构建/工具/依赖变动)
- `revert` -> ⏪ (回退)
- `chore`（合并场景）-> 🔀 (分支合并)

### 3. 内容来源（强制执行 Git Diff 验证）

**🚨 必须先执行 `git diff --name-only HEAD` 和 `git diff HEAD --stat` 查看实际变更的文件**

**关键要求**：
- **明确范围**：`HEAD` 代表上一次 commit，所以 diff 显示的是"从上次提交到现在的所有变更"，**不是**本次对话中的修改
- **严禁仅根据对话上下文猜测或总结变更内容**
- 根据 diff 结果，逐一分析每个变更文件的实际修改内容
- 如果 diff 显示的变更与对话上下文不一致，以 diff 结果为准
- **只描述最终实现的功能，不要包含探索过程中的失败尝试或被放弃的方案**
- 如果变更较为复杂，请在 Header 之后生成详细的 Body 描述（Markdown 列表形式）

**调试代码检查**：
- 在生成 commit 信息前，主动检查 diff 中是否包含**新增的**调试日志（如 `print("🔬")`、`debugLog` 等）
- 如有新增调试日志则主动询问用户是否需要清理
- **如果用户说"清理调试日志，给出提交信息"**：
  - 先执行调试日志清理
  - 然后基于清理后的 diff 生成 commit 信息
  - commit 信息中不提及"清理调试日志"这个动作
- **如果 diff 中只是删除调试日志（清理操作），不要在 commit 信息中提及**
- 调试日志清理是正常的代码清理工作，不是功能性变更，不应占据 commit 信息篇幅

### 4. 输出形式

只输出**一个代码块**，包含完整的 commit 信息（Header + Body）。

不要再额外拆分 Summary 和 Description；如果用户需要单独字段，会明确提出。

**输出后必须主动询问用户是否执行提交**（如"确认无误，是否执行提交？"），不要等用户主动说"执行"。

**禁止**以 `git commit -m "..."` 命令格式输出，只输出纯 commit 信息内容。

commit 信息除了 type 和 scope 部分可以使用英文，其余部分需使用简体中文书写

### 5. 输出示例

```
feat(reader): ✨ 添加阅读进度同步功能

- 新增 ReadingProgressService 处理进度同步逻辑
- 在 ReaderView 中集成进度保存触发点
- 支持 iCloud 跨设备同步
```

### 6. 禁止事项

- 不要提及"移除"、"删除"等在本次提交前不存在的功能或组件
- 不要描述探索过程中的错误路径或失败尝试
- 不要包含调试信息、临时代码等开发过程中的中间状态
- **不要提及调试日志的清理**（如"移除 print 语句"、"清理调试代码"等）
- **不要提及代码格式调整**（如"调整空白行"、"格式化代码"等）
- **严禁不查看 git diff 就直接根据对话内容生成 commit 信息**
- **严禁遗漏 diff 中显示的功能性文件变更**
- 只描述最终交付的功能和改进
- **commit 信息必须与 git diff 中的功能性变更完全对应**
- **禁止附加任何辅助编程标识信息**（如 Co-Authored-By、AI 工具署名等），提交内容必须与展示给用户审核的内容完全一致

## Commit 信息增强规范

对于**性能优化**或**重大功能改进**类型的提交，在 Body 部分增加"优化效果"小节。

### 格式示例

```
⚡️ perf(list): 优化生词库/语境库列表加载性能

- 缓存排序结果为 @State 变量，避免计算属性重复计算
- 创建 LazyView 包装器实现详情页懒加载
- 预计算到期时间缓存，将排序时 5000+ 次查询降为 300 次
- 修复 LazyView 使用 @autoclosure 导致状态捕获错误的 bug

优化效果：
- 生词库加载时间从 7-8 秒降至 1 秒内
- 语境库加载时间从 2-3 秒降至秒进
- 点击列表项能正确进入对应详情页
```

### 适用场景

- `perf` 类型：性能优化提交
- `feat` 类型：涉及用户体验改进的新功能
- `fix` 类型：修复影响用户体验的严重 Bug

### 优化效果描述规范

- 使用**对比形式**：从 X 降至 Y、从 A 改善到 B
- 使用**具体数据**：时间、次数、百分比等可量化指标
- 使用**用户感知语言**：秒进、流畅、无卡顿等

## Merge Commit 专用规范

### 1. Type 与 Emoji

分支合并统一使用 `chore: 🔀`，格式为：

`chore: 🔀 合并 <源分支> 到 <目标分支>，<概括合并内容>`

### 2. Diff 命令差异

**🚨 merge 状态下禁止使用 `git diff HEAD`**，因为 HEAD 仍指向合并前的 commit，会包含当前分支自身的历史变更。

必须使用：
- `git diff --cached --stat`：查看合并引入的变更统计
- `git diff --cached --name-only`：查看合并引入的文件列表

### 3. Body 内容

- 逐一分析 `git diff --cached` 中每个变更文件的实际修改内容
- 按功能模块分组描述，而非逐文件罗列
- 重点描述合并引入的新功能和修复，不需要描述项目配置文件（如 pbxproj）的变更

### 4. 输出形式

与普通 commit 相同，只输出完整版一个代码块。

### 5. 输出示例

```
chore: 🔀 合并 main 分支到 feature/vocabulary-planet，同步滚动阅读器及手势优化

- 新增 WebViewScrollReader 滚动阅读模式
- 优化手势识别，修复滚动模式右滑状态泄漏及边缘不回弹问题
- WebViewPageTurnReader 新增锚点跳转与仿真翻页渲染修复
- WebViewReaderView 新增进度字符偏移量精确恢复
```
