# 任务档案 Skill（分阶段开发 · 跨会话续接）

## 解决的问题

大任务一个会话做不完，换会话常丢进度。用「一个任务 = 一份从开工持续更新到收尾的活档案」来稳定续接。

## 档案位置

- 目录：`docs/tasks/`，命名 `<任务slug>.md`（如 `phrase-lookup.md`）
- 一个大任务一份，全程更新同一份，**不再按日期新建快照**

## 触发点一：开工评估（分析开发思路时自动触发）★核心

用户描述新功能 / 开发需求、你在给方案时，**先做一次分阶段评估**。

判定「需分阶段·建档案」的信号（满足任一）：

- 涉及 ≥3 个模块或文件面广
- 预计一个会话上下文做不完
- 有明确先后依赖（先模型 → 服务 → UI → 联调）
- 用户说「大功能 / 大改造 / 重构」

**判定需分阶段时**：

1. 主动提示：「这任务较大，建议用任务档案分阶段推进，避免跨会话丢进度」
2. 给出**阶段拆分草案**（每阶段：目标 / 涉及文件 / 验收点 / 依赖）
3. 用户确认后在 `docs/tasks/<slug>.md` 建档案（按下方模板）
4. 遵循 protocol-dev：档案是「方案」，写代码仍需逐阶段授权

**判定是小任务时**：不打扰，正常走 protocol-dev。

## 触发点二：交接（开新会话 / 上下文快满 / 说"交接"时自动触发）

无需用户详述，立即：

1. 回顾本会话进度
2. **更新档案「进度台账」+ 勾选「分阶段计划」**（完成阶段打 `[x]`，记 commit SHA、改动文件、下一步起点）
3. 更新 frontmatter 状态 / 更新日期
4. 产出**可复制续接启动词**（独立代码块）：

   ```
   读 @AGENTS.md 协议 + @docs/tasks/<slug>.md 任务档案，按档案「新会话启动指令」续接开发。
   ```

## 触发点三：续接（说"继续 X" / 粘贴启动词时自动触发）

1. 读 `docs/tasks/<slug>.md` 及其指向的关键文件 / 对标文档
2. **复述**：当前第几阶段、上次停在哪、下一步做什么
3. 等授权再动手（protocol-dev）

## 档案模板（写入 `docs/tasks/<slug>.md`）

```markdown
# 任务档案：<名称>

> 状态：进行中 / 已完成｜分支：`feat/xxx`｜创建：YYYY-MM-DD｜更新：YYYY-MM-DD

## 0. 新会话启动指令（AI 必读）
1. 读 `@AGENTS.md` 开发协议
2. 读本档案全文
3. 复述「当前阶段 + 上次停点 + 下一步」，等授权，禁止直接动手

## 1. 目标与验收标准
- 目标：一句话
- 验收：如何判定整体完成

## 2. 分阶段计划
- [x] 阶段1 <名>：<目标> — 完成（commit `abc1234`）
- [ ] 阶段2 <名>：<目标> — 待做｜依赖：阶段1
- [ ] 阶段3 …

## 3. 架构与上下文
- 关键文件：`路径` — 作用
- 技术决策：…
- 参考文档：`路径` — 相关设计 / 技术文档

## 4. 进度台账（每次会话末追加，倒序）
### YYYY-MM-DD（会话N）
- 做了：…
- commit：`SHA` <msg>
- 下一步：从 `文件:行` 起，做 …
- 遗留/坑：…

## 5. 决策与坑记录
- …
```

## 与其他机制的分工（避免重复）

- **protocol-dev**：开工评估 = protocol-dev「方案设计」的加强版；写代码仍走其逐步授权。
- **memory `*-progress.md`**：退化为**一行指针**指向档案，不再两处重复写进度明细。
- **旧交接快照**：一律被活档案取代；新任务统一用 `docs/tasks/`，全程更新同一份。

## 强制规范

- 档案写入/更新前**先展示、等确认**（先谋后动）。
- 台账 append-only，勾选可改，不删历史。
- 一个任务只用一份档案，全程更新它。
